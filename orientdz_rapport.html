Résumé

🇫🇷 Français

Le présent mémoire décrit la conception et le développement d'OrientDZ, une plateforme mobile d'orientation médicale bilingue (français/arabe) destinée au contexte algérien. Face à la fragmentation des ressources de santé numériques et aux barrières linguistiques persistantes, OrientDZ propose une architecture modulaire intégrant deux composantes majeures : OrientNutri, un système expert de calcul nutritionnel entièrement décentralisé fonctionnant sans recours à des API cloud, et SauveMoi QR, un module de sécurité d'urgence médicale 100% hors-ligne reposant sur un protocole de chiffrement local XOR combiné à l'encodage Base64.

Le moteur nutritionnel implémente les équations de Mifflin-St Jeor pour le calcul du métabolisme basal (BMR), applique les facteurs d'activité physique standardisés pour dériver le TDEE (Total Daily Energy Expenditure), puis distribue les macronutriments selon des ratios scientifiquement validés et personnalisés selon l'objectif de l'utilisateur. L'intégralité des calculs s'effectue localement sur l'appareil, garantissant une totale indépendance vis-à-vis de la connectivité réseau.

Le module SauveMoi QR génère une fiche clinique d'urgence chiffrée, encodée dans un QR code scannable par les secouristes, même en l'absence totale de connexion internet. La gestion du bilinguisme natif français/arabe est assurée par un système de localisation basé sur le stockage local (localStorage/AsyncStorage), sans nécessiter de backend.

Mots-clés : santé mobile, orientation médicale, système expert, TDEE, macronutriments, chiffrement XOR, Base64, QR code médical, offline, bilinguisme, Algérie.
🇬🇧 Abstract (English)

This thesis presents the design and development of OrientDZ, a bilingual (French/Arabic) mobile medical orientation platform tailored to the Algerian healthcare context. OrientDZ addresses fragmented digital health resources through a modular architecture integrating two key components: OrientNutri, a fully decentralized nutritional expert system operating without cloud APIs, and SauveMoi QR, a 100% offline medical emergency module based on local XOR encryption combined with Base64 encoding.

The nutritional engine implements Mifflin-St Jeor equations for Basal Metabolic Rate (BMR) computation, applies standardized physical activity factors to derive the Total Daily Energy Expenditure (TDEE), and distributes macronutrients according to scientifically validated ratios personalized to user goals. All computations run locally on the device, ensuring complete network independence. The SauveMoi QR module generates an encrypted emergency clinical card, embedded in a QR code readable by first responders without internet connectivity.

Keywords: mobile health, medical orientation, expert system, TDEE, macronutrients, XOR encryption, Base64, medical QR code, offline, bilingualism, Algeria.

Introduction Générale

1. Contexte général : la digitalisation de la santé en Algérie

La transformation numérique du secteur de la santé constitue, à l'échelle mondiale, l'un des enjeux les plus structurants de ce début de siècle. Les technologies de l'information et de la communication (TIC) redéfinissent profondément la relation entre le patient et le système de soins, en offrant des outils d'aide à la décision, d'orientation médicale, de suivi thérapeutique et de communication d'urgence qui transcendent les contraintes géographiques et socio-économiques traditionnelles.

En Algérie, ce mouvement de digitalisation sanitaire se heurte à des réalités complexes. D'un côté, le pays dispose d'une infrastructure télécoms en expansion rapide, d'un taux de pénétration du smartphone supérieur à 60% en milieu urbain et d'une population jeune fortement connectée. De l'autre, le système de santé public reste confronté à des défis structurels majeurs : la surcharge des services hospitaliers, l'inégale répartition géographique des ressources médicales, l'absence d'un référentiel numérique unifié d'orientation des patients et la fracture linguistique entre les patients arabophones et un système de santé dont la documentation demeure largement francophone.

La pandémie de COVID-19 (2020-2021) a agi comme un catalyseur brutal de cette transition numérique, révélant en même temps les limites d'un système de santé peu préparé à gérer des flux massifs d'information médicale en temps réel et à distance. Les applications de télémédecine et les outils d'orientation mobile ont alors montré leur valeur ajoutée, mais également leurs lacunes : dépendance aux serveurs distants, absence de fonctionnement hors-ligne, barrières linguistiques non traitées et vulnérabilité des données personnelles de santé.

2. Problématique

C'est dans ce contexte qu'émerge la problématique centrale à laquelle s'attaque le projet OrientDZ. Plusieurs constats factuels ont guidé la formulation de cette problématique :

  • Fragmentation de l'information médicale : il n'existe pas, à ce jour, de plateforme mobile unifiée permettant à un citoyen algérien d'accéder simultanément à des informations d'orientation médicale, à un suivi nutritionnel personnalisé et à un dispositif d'urgence médicale, le tout sans connexion internet permanente.
  • Barrière linguistique : la majorité des applications de santé disponibles sur les marchés numériques mondiaux ne proposent pas de support natif en arabe adapté au dialecte et aux conventions médicales algériennes. Le passage d'une langue à l'autre implique, dans la quasi-totalité des solutions existantes, une architecture applicative distincte ou une dépendance à des services de traduction cloud.
  • Vulnérabilité des données médicales en situation d'urgence : lorsqu'un patient se retrouve inconscient ou dans l'incapacité de communiquer verbalement, les secouristes n'ont généralement aucun moyen rapide et fiable d'accéder à ses informations médicales critiques (groupe sanguin, allergies, traitements en cours, pathologies chroniques). Les solutions existantes nécessitent soit une connexion internet, soit un accès à des serveurs distants, rendant leur utilisation aléatoire dans les zones à faible couverture réseau.
  • Dépendance excessive aux infrastructures cloud : la majorité des applications de santé mobiles reposent sur des architectures client-serveur nécessitant une connexion permanente, ce qui les rend inutilisables dans des zones à connectivité réduite, fréquentes en dehors des grandes agglomérations algériennes.

Ces constats convergent vers une question de recherche principale : comment concevoir une plateforme mobile de santé, adaptée au contexte algérien, qui soit à la fois bilingue (FR/AR) de manière native, indépendante des infrastructures cloud pour ses fonctions critiques, et capable d'assurer la protection des données médicales privées même en l'absence totale de connectivité ?

3. Objectifs du projet

OrientDZ se fixe pour objectif de répondre à cette problématique au travers d'une architecture logicielle mobile modulaire, sobre et résiliente. Les objectifs spécifiques du projet peuvent être déclinés comme suit :

3.1 Objectifs fonctionnels

  • Concevoir et implémenter un système expert nutritionnel décentralisé (OrientNutri) calculant le métabolisme basal (BMR) et la dépense énergétique journalière totale (TDEE) selon les équations de Mifflin-St Jeor, et distribuant les macronutriments (protéines, lipides, glucides) selon des ratios validés scientifiquement, sans recours à aucune API cloud.
  • Développer un module de sécurité d'urgence médicale (SauveMoi QR) permettant la saisie, le chiffrement local (XOR + Base64) et la restitution graphique sous forme de QR Code d'une fiche clinique d'urgence, accessible par les secouristes sans connexion internet.
  • Implémenter un système de localisation bilingue natif (français/arabe) reposant exclusivement sur le stockage local de l'appareil (localStorage/AsyncStorage), sans appel à des services de traduction distants.
  • Assurer une expérience utilisateur cohérente et accessible, y compris l'adaptation graphique pour la langue arabe (interface RTL — de droite à gauche).

3.2 Objectifs architecturaux

  • Adopter une architecture hors-ligne par défaut (offline-first), dans laquelle toutes les fonctionnalités critiques restent opérationnelles sans connexion réseau.
  • Garantir la décentralisation des données : aucune donnée médicale personnelle ne transite vers des serveurs distants.
  • Concevoir une architecture modulaire permettant l'ajout ou la suppression de composants sans déstabiliser le cœur applicatif.

3.3 Objectifs académiques et de recherche

  • Démontrer la faisabilité d'un système expert médical embarqué sur mobile sans infrastructure serveur.
  • Évaluer les protocoles de chiffrement légers adaptés aux contraintes des appareils mobiles pour la protection des données médicales.
  • Proposer un modèle de localisation bilingue natif applicable à d'autres domaines applicatifs du contexte algérien.

4. Périmètre et décisions de conception

Le projet a connu, au cours de sa phase d'analyse, une évolution significative de son périmètre initial. Un troisième module, initialement intitulé PharmaInfo, avait été envisagé dans le cahier des charges préliminaire. Il était destiné à fournir des informations sur les médicaments disponibles en Algérie, leurs génériques, leurs indications et contre-indications. Après une analyse approfondie des contraintes réglementaires liées à la diffusion d'informations pharmaceutiques, des risques liés à l'automédication que pourrait induire une telle fonctionnalité, et des implications en termes de responsabilité médicale, il a été décidé de retirer définitivement ce module du périmètre du projet.

Cette décision a permis de recentrer l'effort de développement sur les deux modules à plus forte valeur ajoutée technique et à plus faible risque : OrientNutri et SauveMoi QR, dont les spécifications sont présentées en détail dans les chapitres qui suivent.

5. Méthodologie de travail

La conduite du projet OrientDZ a suivi une démarche itérative et incrémentale, inspirée des principes Agile adaptés au contexte académique. Le travail a été organisé en trois phases principales :

  • Phase d'analyse et de spécification (4 semaines) : recueil et formalisation des besoins, étude de l'existant, définition des contraintes architecturales et décisions de périmètre.
  • Phase de conception et d'implémentation (8 semaines) : design de l'architecture, implémentation itérative des deux modules, développement de l'interface bilingue.
  • Phase de test et de validation (3 semaines) : tests unitaires des algorithmes nutritionnels, tests de conformité du chiffrement, tests d'interface utilisateur et validation des scénarios d'urgence.

6. Structure du rapport

Le présent mémoire est structuré en trois chapitres techniques, encadrés par cette introduction et une conclusion générale.

  • Le Chapitre 1 présente l'architecture globale du système OrientDZ, les spécifications techniques, les technologies employées et le système de localisation bilingue.
  • Le Chapitre 2 détaille le système expert nutritionnel OrientNutri : fondements théoriques, algorithmes de calcul du TDEE et des macronutriments, et implémentation bilingue.
  • Le Chapitre 3 expose le module de sécurité SauveMoi QR : protocole de chiffrement XOR + Base64, fonctionnement offline, structure de la fiche clinique et génération du QR code médical.
  • La Conclusion générale synthétise les contributions du projet et ouvre des perspectives d'évolution, notamment dans le cadre d'une valorisation sous forme de startup.

1.1 Introduction

Ce chapitre constitue le socle architectural de l'ensemble du mémoire. Il décrit les décisions de conception fondamentales du projet OrientDZ, depuis l'analyse des besoins jusqu'à l'architecture modulaire retenue, en passant par les technologies employées et le système de gestion bilingue.

Toute architecture logicielle repose sur un ensemble de choix qui engagent durablement la qualité, la maintenabilité et l'évolutivité d'un système. Dans le cas d'OrientDZ, ces choix ont été particulièrement déterminants, car ils conditionnent non seulement les performances fonctionnelles de la plateforme, mais également sa capacité à opérer dans des conditions dégradées de connectivité, à respecter la vie privée des utilisateurs et à s'adapter à deux systèmes d'écriture de directions opposées.

1.2 Présentation et genèse du projet OrientDZ

OrientDZ est une plateforme mobile d'orientation médicale destinée aux citoyens algériens. Son nom combine la dimension d'orientation — au sens médical du terme, c'est-à-dire l'aide à la navigation dans le système de soins — et le code ISO 3166-1 alpha-2 de l'Algérie (DZ), affirmant ainsi son ancrage territorial et culturel spécifique.

La genèse du projet remonte à l'observation d'un vide applicatif significatif dans l'écosystème numérique de santé algérien. Si des applications généralistes de santé existent sur les marchés mondiaux, elles ne tiennent pas compte des spécificités algériennes : système de santé à deux vitesses (public/privé), distribution géographique des spécialistes, barrière linguistique français-arabe, et accès réseau irrégulier en dehors des grands centres urbains.

Le projet se fixe comme ambition de proposer un outil concret, opérationnel et immédiatement utile pour trois profils d'utilisateurs distincts :

  • Le patient ordinaire cherchant à comprendre ses besoins nutritionnels ou à conserver ses informations médicales de manière sécurisée.
  • La personne atteinte d'une pathologie chronique (diabète, hypertension, allergie grave) nécessitant que ses informations médicales critiques soient instantanément accessibles en cas d'urgence.
  • Le professionnel de santé ou secouriste ayant besoin d'accéder rapidement à la fiche médicale d'un patient inconscient sans avoir à contacter un serveur distant.

1.4 Décisions architecturales — retrait du module PharmaInfo

La phase d'analyse a conduit à une décision architecturale majeure : le retrait définitif du module PharmaInfo du périmètre fonctionnel d'OrientDZ. Ce module, initialement planifié pour fournir un catalogue interactif de médicaments disponibles sur le marché algérien avec leurs indications, posologies et contre-indications, a été écarté pour plusieurs raisons déterminantes.

Motivation du retrait de PharmaInfo

Contraintes réglementaires : la diffusion d'informations pharmaceutiques sur une application mobile non homologuée par les autorités sanitaires algériennes (Agence Nationale des Produits Pharmaceutiques) exposerait le projet à des risques juridiques significatifs.

Risque d'automédication : la mise à disposition directe d'une base de données médicamenteuse sans encadrement professionnel pourrait encourager des comportements d'automédication potentiellement dangereux.

Maintenance de la base de données : le catalogue des médicaments disponibles en Algérie est soumis à des mises à jour fréquentes liées aux ruptures d'approvisionnement et aux changements de politique d'importation, nécessitant une infrastructure de mise à jour continue incompatible avec l'approche offline-first.

Recentrage stratégique : le retrait de PharmaInfo a permis de concentrer les ressources de développement sur OrientNutri et SauveMoi QR, dont la valeur ajoutée technique et sociale est jugée plus élevée et le cadre réglementaire moins contraignant.

Cette décision illustre la maturité de la démarche de conception adoptée : la capacité à identifier les risques en amont et à ajuster le périmètre en conséquence est une compétence d'ingénierie fondamentale, particulièrement critique dans le domaine de la santé numérique où les conséquences d'une fonctionnalité mal conçue peuvent dépasser le cadre purement technique.

1.5 Architecture modulaire retenue

L'architecture d'OrientDZ suit le patron de conception modulaire à couplage faible (loosely coupled modular architecture). Chaque module fonctionnel constitue une unité autonome disposant de son propre espace de données local, de sa propre logique métier et de son propre ensemble de composants d'interface. Les modules communiquent via un état applicatif global partagé, mais ne partagent pas de données sensibles entre eux.

2.2 État de l'art en nutrition computationnelle

2.2.1 Évolution des équations d'estimation du métabolisme basal

L'estimation du métabolisme basal (BMR — Basal Metabolic Rate) à partir de données anthropométriques a fait l'objet d'une évolution scientifique continue depuis la fin du XIXe siècle. Trois équations majeures jalonnent cette progression :

L'équation de Harris-Benedict (1919) constitue historiquement la première tentative de formalisation mathématique du BMR à partir du poids, de la taille et de l'âge. Révisée par Roza et Shizgal en 1984, elle a longtemps été la référence clinique :

BMRhomme = 88,362 + (13,397 × P) + (4,799 × T) − (5,677 × A) BMRfemme = 447,593 + (9,247 × P) + (3,098 × T) − (4,330 × A) P = poids (kg), T = taille (cm), A = âge (années) — Harris-Benedict révisée

L'équation de Mifflin-St Jeor (1990) a été développée par Mifflin et al. à l'issue d'une étude portant sur 498 sujets. Une méta-analyse ultérieure publiée dans le Journal of the American Dietetic Association (Frankenfield et al., 2005) a établi que cette équation présente une précision supérieure à celle de Harris-Benedict, avec une erreur moyenne de prédiction inférieure à 10% pour la majorité des individus. C'est pour cette raison qu'elle a été retenue comme base algorithmique d'OrientNutri :

BMRhomme = (10 × P) + (6,25 × T) − (5 × A) + 5 BMRfemme = (10 × P) + (6,25 × T) − (5 × A) − 161 Équation de Mifflin-St Jeor — retenue dans OrientNutri

L'équation de Katch-McArdle, plus récente, intègre la masse musculaire maigre et présente une précision supérieure pour les individus athlétiques. Cependant, sa mise en œuvre nécessite une mesure du pourcentage de masse grasse que l'utilisateur ne peut généralement pas fournir avec fiabilité sans équipement spécialisé. Pour des raisons de praticité et d'accessibilité, elle n'a pas été retenue dans la version actuelle d'OrientNutri.

2.2.2 Systèmes experts nutritionnels — revue des solutions existantes

  • MyFitnessPal (Under Armour) : application internationale très complète, mais entièrement dépendante d'une connexion internet pour ses calculs et son catalogue d'aliments. Aucun support de l'arabe.
  • Cronometer : outil nutritionnel précis, orienté utilisateurs avancés, sans support arabe et nécessitant un compte utilisateur sur des serveurs distants.

Aucune des solutions existantes ne satisfait simultanément les trois exigences fondamentales d'OrientNutri : fonctionnement hors-ligne, support natif de l'arabe et absence de transmission de données vers des serveurs tiers.

2.4 Modèle mathématique — Calcul du Métabolisme de Base (BMR)

2.4.1 Fondements de l'équation de Mifflin-St Jeor

L'équation de Mifflin-St Jeor modélise la dépense énergétique d'un organisme au repos complet — soit l'énergie minimale requise pour maintenir les fonctions vitales (respiration, circulation sanguine, thermorégulation, activité cellulaire). Elle intègre quatre paramètres anthropométriques facilement mesurables par l'utilisateur : le poids corporel en kilogrammes, la taille en centimètres, l'âge en années et le sexe biologique.

Le différentiel de constante entre les deux formules sexuées (+5 pour les hommes, -161 pour les femmes) reflète les différences physiologiques moyennes de composition corporelle, notamment le ratio masse musculaire / masse grasse, qui influence directement le taux métabolique au repos.

2.4.2 Implémentation TypeScript du calcul du BMR

Algorithme 2.1 — Calcul du BMR (Mifflin-St Jeor) en TypeScript
// Types et interfaces type Genre = 'homme' | 'femme'; interface DonneesAnthropo { poids: number; // kg taille: number; // cm age: number; // années genre: Genre; } function calculerBMR(data: DonneesAnthropo): number { // Validation des plages physiologiques if (data.poids < 20 || data.poids > 300) throw new Error('Poids hors plage'); if (data.taille < 100 || data.taille > 250) throw new Error('Taille hors plage'); if (data.age < 15 || data.age > 100) throw new Error('Âge hors plage'); const base = (10 * data.poids) + (6.25 * data.taille) - (5 * data.age); const constante = data.genre === 'homme' ? 5 : -161; return Math.round(base + constante); }

2.5 Dérivation du TDEE (Total Daily Energy Expenditure)

2.5.1 Le Niveau d'Activité Physique (NAP)

Le BMR ne représente que la dépense énergétique au repos complet. Pour estimer les besoins caloriques réels d'un individu dans sa vie quotidienne, il est nécessaire de multiplier le BMR par un facteur correctif appelé Niveau d'Activité Physique (NAP), parfois désigné par l'acronyme anglais Physical Activity Level (PAL). Ce facteur, établi empiriquement par des études de doublement de l'eau deutérée (doubly labeled water), représente le rapport entre la dépense énergétique totale et le métabolisme basal.

2.5.2 Ajustement selon l'objectif de l'utilisateur

Le TDEE brut représente la quantité de calories nécessaire pour maintenir le poids corporel actuel. OrientNutri propose trois objectifs distincts qui modifient ce TDEE de référence :

  • Perte de poids (déficit calorique) : le TDEE ajusté est réduit d'un déficit de 500 kcal/jour, ce qui correspond théoriquement à une perte de masse corporelle d'environ 0,5 kg par semaine, valeur reconnue comme sûre et soutenable par les recommandations de l'OMS.
  • Maintien du poids : le TDEE ajusté est égal au TDEE brut. L'apport calorique est conçu pour équilibrer précisément la dépense.
  • Prise de masse musculaire (surplus calorique) : le TDEE ajusté est augmenté d'un surplus de 300 kcal/jour, correspondant à un surplus modéré favorisant la synthèse protéique sans accumulation excessive de masse grasse.

2.6.1 Fondements de la distribution macronutritionnelle

3.1 Introduction et problématique

Les situations médicales d'urgence se caractérisent par leur imprévisibilité et par la nécessité absolue d'un accès immédiat aux informations cliniques vitales du patient. Un individu inconscient, en état de choc ou incapable de communiquer ne peut transmettre verbalement ses antécédents médicaux, ses allergies aux médicaments ou son groupe sanguin aux personnels de secours. Cette lacune informationnelle peut entraîner des décisions thérapeutiques sous-optimales, voire dangereuses.

La réponse conventionnelle à ce problème passe par des bracelets médicaux, des cartes plastifiées ou des formulaires papier — des supports physiques facilement perdus, oubliés, ou illisibles dans des conditions de terrain dégradées. Le déploiement de solutions numériques centralisées (serveurs hospitaliers, bases de données nationales) soulève quant à lui des problèmes d'accès hors-ligne, de confidentialité et de dépendance infrastructurelle.

SauveMoi QR répond à cette problématique en proposant une fiche clinique d'urgence numérique, stockée localement sur l'appareil de l'utilisateur et encodée dans un QR code généré offline. Ce QR code, lisible par n'importe quel scanner standard, contient les données médicales essentielles protégées par un chiffrement XOR combiné à un encodage Base64. 3.4 Protocole de chiffrement XOR + Base64

3.4.1 Fondements mathématiques du chiffrement XOR

Le chiffrement XOR (eXclusive OR) est une opération logique binaire dont la propriété fondamentale d'involution le rend particulièrement adapté à la cryptographie symétrique légère. La base mathématique repose sur trois propriétés algébriques :

Propriété 1 — Involution :

A ⊕ B ⊕ B = A

Propriété 2 — Commutativité : A ⊕ B = B ⊕ A

Propriété 3 — Associativité : (A ⊕ B) ⊕ C = A ⊕ (B ⊕ C)

Conséquence directe : La même fonction réalise le chiffrement et le déchiffrement — decrypt(key, encrypt(key, data)) = data

Dans SauveMoi QR, la clé de chiffrement est dérivée d'un code PIN à 6 chiffres fourni par l'utilisateur. Cette clé PIN est étendue à la longueur des données à chiffrer par répétition cyclique, garantissant que chaque octet de données est XORé avec un octet de clé.

3.4.2 Pipeline complet de chiffrement

Le chiffrement s'effectue en trois étapes successives :

TypeScript — Implémentation du chiffrement XOR + Base64
function chiffrerDonnees(donnees: string, pinCode: string): string { // Étape 1 : Conversion des données et de la clé en tableaux d'octets const encoder = new TextEncoder(); const octetesDonnees = encoder.encode(donnees); const octetesCle = encoder.encode(pinCode); const longueurCle = octetesCle.length; // Étape 2 : Application du XOR octet par octet (clé cyclique) const resultat = new Uint8Array(octetesDonnees.length); for (let i = 0; i < octetesDonnees.length; i++) { resultat[i] = octetesDonnees[i] ^ octetesCle[i % longueurCle]; } // Étape 3 : Encodage Base64 pour compatibilité QR code (ASCII-safe) return btoa(String.fromCharCode(...resultat)); } function dechiffrerDonnees(payloadBase64: string, pinCode: string): string { // Décodage Base64 → octets chiffrés const octetsChiffres = Uint8Array.from(atob(payloadBase64), c => c.charCodeAt(0)); const octetesCle = new TextEncoder().encode(pinCode); const longueurCle = octetesCle.length; // XOR inverse (identique au chiffrement par involution) const resultat = new Uint8Array(octetsChiffres.length); for (let i = 0; i < octetsChiffres.length; i++) { resultat[i] = octetsChiffres[i] ^ octetesCle[i % longueurCle]; } return new TextDecoder().decode(resultat); }

3.4.3 Rôle de l'encodage Base64

Le résultat brut d'une opération XOR sur des octets UTF-8 produit des valeurs binaires arbitraires (octets 0x00–0xFF), dont beaucoup sont des caractères de contrôle non imprimables incompatibles avec les encodages QR code standard. L'encodage Base64 transforme ce flux binaire en une chaîne de 64 caractères ASCII imprimables (A–Z, a–z, 0–9, +, /), garantissant la compatibilité avec tous les générateurs et lecteurs de QR codes.

3.8 Analyse de sécurité et limites

3.8.1 Forces de sécurité

  • Aucune surface d'attaque réseau : l'absence totale de communications serveur élimine les vecteurs d'attaque les plus communs (interception TLS, injections API, fuites de base de données).
  • Principe du moindre privilège : l'application ne demande que les permissions caméra (scan QR) et stockage local, sans accès aux contacts, localisation ou réseau.
  • Données chiffrées au repos : même un accès physique non autorisé à l'appareil ne permet pas de lire les données médicales sans le PIN.
  • Transparence algorithmique : XOR et Base64 sont des algorithmes publics, documentés et vérifiables, contrairement à des implémentations propriétaires opaques.

3.8.2 Limites connues et perspectives

Il est important d'énoncer honnêtement les limites de l'approche XOR dans un contexte académique :

  • Vulnérabilité aux attaques par texte clair connu : si un attaquant connaît une partie du JSON en clair (ex : les champs fixes comme "versionSchema":1), il peut en déduire des portions de la clé XOR. Cette vulnérabilité est atténuée par la longueur variable des données et la nature privée du PIN.
  • Force de la clé limitée : un PIN à 6 chiffres offre 106 = 1 000 000 combinaisons, susceptibles d'une attaque par force brute exhaustive. Une version future pourrait adopter AES-256-GCM via la bibliothèque react-native-crypto.
  • Dépendance à la mémorisation du PIN : si l'utilisateur oublie son PIN, les données ne sont pas récupérables — ce qui est une garantie de confidentialité, mais aussi un risque de perte d'accès.

Limites du travail

Une démarche scientifique honnête impose de reconnaître les limites de la réalisation présentée :

  • Robustesse cryptographique limitée de XOR : comme discuté au Chapitre 3, l'algorithme XOR est vulnérable aux attaques par texte clair connu. Il convient au niveau de protection requis pour un usage grand public, mais ne satisferait pas les exigences d'une certification médicale de type CE (marquage européen des dispositifs médicaux).
  • Absence d'évaluation clinique externe : les recommandations nutritionnelles d'OrientNutri sont basées sur des équations validées, mais n'ont pas fait l'objet d'une évaluation par des nutritionnistes ou diététiciens diplômés dans le cadre de ce mémoire.

  • Tests utilisateurs limités : les tests ont été conduits sur un panel restreint ; une étude UX formelle avec des utilisateurs représentatifs (alphabétisation variable, familiarité avec les smartphones) renforcerait la validation ergonomique.

Annexe A — Implémentation complète du moteur de chiffrement XOR + Base64

Cette annexe présente le code source complet du module de chiffrement de SauveMoi QR, incluant la gestion des erreurs, la validation de la clé PIN et les fonctions utilitaires associées.

sauvemoi/crypto/xorCipher.ts — Module de chiffrement complet
/** * Module de chiffrement XOR + Base64 pour SauveMoi QR * Algorithme : XOR symétrique avec clé cyclique, encodage Base64 pour QR */ import { FicheUrgence } from '../types/FicheUrgence'; const LONGUEUR_PIN_MIN = 4; const LONGUEUR_PIN_MAX = 12; export function validerPin(pin: string): boolean { return /^\d+$/.test(pin) && pin.length >= LONGUEUR_PIN_MIN && pin.length <= LONGUEUR_PIN_MAX; } function xorBuffer(data: Uint8Array, key: Uint8Array): Uint8Array { const result = new Uint8Array(data.length); for (let i = 0; i < data.length; i++) { result[i] = data[i] ^ key[i % key.length]; } return result; } export function chiffrerFiche(fiche: FicheUrgence, pin: string): string { if (!validerPin(pin)) throw new Error('PIN invalide : 4 à 12 chiffres requis'); const encoder = new TextEncoder(); const dataBytes = encoder.encode(JSON.stringify(fiche)); const keyBytes = encoder.encode(pin); const xored = xorBuffer(dataBytes, keyBytes); return btoa(String.fromCharCode(...xored)); } export function dechiffrerFiche(payload: string, pin: string): FicheUrgence { if (!validerPin(pin)) throw new Error('PIN invalide'); const encrypted = Uint8Array.from(atob(payload), c => c.charCodeAt(0)); const keyBytes = new TextEncoder().encode(pin); const decrypted = xorBuffer(encrypted, keyBytes); const jsonStr = new TextDecoder().decode(decrypted); try { return JSON.parse(jsonStr) as FicheUrgence; } catch { throw new Error('PIN incorrect ou données corrompues'); } }
